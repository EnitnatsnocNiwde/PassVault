# PassVault 坚果云同步规则设计

## 1. 设计目标

PassVault 是一个密码管理箱，本地文件和云端文件都属于高价值数据。同步逻辑必须优先保证数据安全，避免因为版本号判断错误、密钥不一致、误删、覆盖冲突等问题导致用户数据丢失。

因此，同步策略不能只依赖版本号。版本号只能用于判断新旧，不能单独决定上传、下载或覆盖。

同步逻辑需要同时判断以下信息：

1. 本地文件是否存在
2. 云端文件是否存在
3. 本地文件和云端文件是否属于同一个密码库
4. 当前密钥是否可以解密本地文件
5. 当前密钥是否可以解密云端文件
6. 本地版本和云端版本的新旧关系
7. 本地是否存在未同步修改
8. 云端是否存在其他设备修改
9. 是否存在大量删除风险
10. 是否需要用户手动确认

设计原则：

* 能安全自动同步的场景，自动同步。
* 有数据丢失风险的场景，必须提示用户。
* 密钥不一致时，禁止自动解析、合并、覆盖。
* 覆盖本地或云端前，尽量先创建备份。
* 默认操作应偏保守，不能默认执行高风险覆盖。
* 用户误删数据时，应该有机会从云端恢复。

---

## 2. 文件元数据设计

本地文件和云端文件建议都包含以下元数据：

```json
{
  "vaultId": "唯一密码库ID",
  "deviceId": "当前设备ID",
  "version": 10,
  "lastSyncVersion": 9,
  "updatedAt": 1710000000000,
  "contentHash": "内容哈希",
  "itemCount": 100,
  "deletedItems": [],
  "encryptedData": "加密后的真实数据"
}
```

字段说明：

| 字段              | 说明                         |
| --------------- | -------------------------- |
| vaultId         | 密码库唯一 ID，用于判断本地和云端是否属于同一个库 |
| deviceId        | 当前设备 ID，用于辅助判断修改来源         |
| version         | 当前数据版本号，每次保存后递增            |
| lastSyncVersion | 本地上一次成功同步时对应的云端版本          |
| updatedAt       | 最后更新时间                     |
| contentHash     | 内容哈希，用于判断内容是否真的变化          |
| itemCount       | 当前密码条目数量，用于检测大量删除          |
| deletedItems    | 删除记录，用于区分正常删除和异常丢失         |
| encryptedData   | 加密后的密码数据                   |

其中 `vaultId` 非常重要。

如果本地和云端的 `vaultId` 不一致，说明它们不是同一个密码库，不能直接用版本号判断谁覆盖谁。

---

## 3. 密钥验证规则

用户输入密钥后，需要分别尝试解密本地文件和云端文件。

结果分为四种：

### 3.1 本地和云端都能解密

说明当前密钥有效，可以继续执行同步判断。

处理：

```text
继续判断 vaultId、版本号、lastSyncVersion、hash 和删除风险。
```

---

### 3.2 本地能解密，云端不能解密

说明当前密钥只能打开本地文件，无法打开云端文件。

可能原因：

1. 云端文件使用的是旧密钥
2. 云端文件属于另一个密码库
3. 当前输入的密钥不是云端文件的密钥
4. 云端文件损坏

处理规则：

```text
禁止自动下载云端。
禁止自动合并云端。
禁止自动覆盖云端。
禁止自动删除云端。
```

需要提示用户：

```text
当前密钥可以打开本地密码库，但无法打开云端同步文件。

可能原因：
1. 云端文件使用的是旧密钥
2. 云端文件属于另一个密码库
3. 当前输入的密钥不是云端文件的密钥
4. 云端文件可能已损坏

请选择：
- 重新输入云端旧密钥
- 使用本地数据覆盖云端
- 先备份云端文件，再使用本地覆盖云端
- 取消同步
```

默认选项应该是：

```text
取消同步
```

或者：

```text
重新输入云端旧密钥
```

不能默认覆盖云端。

---

### 3.3 本地不能解密，云端能解密

说明当前密钥可以打开云端文件，但无法打开本地文件。

可能原因：

1. 本地文件使用的是另一个密钥
2. 本地文件损坏
3. 用户当前想切换到云端库

处理规则：

```text
禁止自动覆盖本地。
禁止自动删除本地。
```

需要提示用户：

```text
当前密钥可以打开云端密码库，但无法打开本地密码库。

请选择：
- 重新输入本地旧密钥
- 备份本地文件后，使用云端文件覆盖本地
- 取消同步
```

如果用户选择使用云端覆盖本地，必须先备份本地原文件。

---

### 3.4 本地和云端都不能解密

说明当前密钥不正确，或者两个文件都损坏。

处理：

```text
提示密钥错误或文件异常。
不执行任何同步操作。
```

提示：

```text
当前密钥无法打开本地文件，也无法打开云端文件。
请确认密钥是否正确。
```

---

## 4. 登录界面的密钥提示规则

在用户登录或选择密钥时，如果已经配置了坚果云同步，应优先提醒用户：

```text
如果你之前已经在云端同步过密码库，请输入当时用于加密云端密码库的密钥。
如果使用新的密钥，可能无法打开旧的云端同步文件。
```

如果用户输入的新密钥无法打开云端文件，不应该强行解析云端文件。

处理：

```text
直接提示云端密钥不匹配。
允许用户重新输入旧密钥。
允许用户选择本地覆盖云端，但必须明确提示风险。
```

---

## 5. 同步入口规则

开始同步时，先判断本地文件和云端文件是否存在。

### 5.1 本地无文件，云端无文件

说明用户第一次使用。

处理：

```text
创建新的本地密码库。
生成新的 vaultId。
初始化 version = 1。
如果用户开启云同步，则上传到云端。
```

---

### 5.2 本地有文件，云端无文件

说明用户本地已有数据，但云端还没有同步文件。

处理：

```text
提示用户是否将本地密码库上传到云端。
```

提示文案：

```text
云端没有发现同步文件。
是否将当前本地密码库上传到云端？
```

用户确认后：

```text
上传本地文件到云端。
设置 lastSyncVersion = local.version。
```

---

### 5.3 本地无文件，云端有文件

说明用户可能在新设备上登录。

处理：

```text
要求用户输入云端文件的密钥。
密钥验证成功后，下载云端文件到本地。
```

下载前提示：

```text
检测到云端已有密码库。
请输入该云端密码库的密钥以恢复数据。
```

---

### 5.4 本地有文件，云端也有文件

这是最复杂的场景。

处理顺序：

```text
1. 验证密钥
2. 判断 vaultId 是否一致
3. 判断 contentHash 是否一致
4. 判断版本号
5. 判断 lastSyncVersion
6. 判断是否存在大量删除
7. 决定自动同步或提示用户
```

---

## 6. vaultId 判断规则

### 6.1 vaultId 相同

说明本地和云端属于同一个密码库。

处理：

```text
继续执行版本判断。
```

---

### 6.2 vaultId 不同

说明本地和云端不是同一个密码库。

不能直接用版本号判断，也不能自动覆盖。

提示用户：

```text
检测到本地密码库和云端密码库不是同一个库。

本地和云端可能是两个不同的密码库。
如果继续覆盖，可能导致其中一份数据丢失。

请选择：
- 使用本地密码库覆盖云端
- 备份本地后，使用云端密码库覆盖本地
- 取消同步
```

默认选项：

```text
取消同步
```

如果用户选择覆盖，必须二次确认。

---

## 7. contentHash 判断规则

如果：

```text
local.contentHash == remote.contentHash
```

说明本地和云端内容完全一致。

处理：

```text
无需同步。
可以更新同步状态。
```

此时即使 version 相同，也没有问题。

如果 hash 不一致，再继续比较版本号。

---

## 8. 版本号判断规则

版本号只表示数据新旧，不能单独决定同步方向。

版本关系分三种：

```text
local.version > remote.version
local.version < remote.version
local.version == remote.version
```

---

## 9. 本地版本大于云端版本

例如：

```text
local.version = 10
remote.version = 9
```

这表示本地看起来比云端新，但不能直接上传。

需要继续判断：

```text
local.lastSyncVersion 是否等于 remote.version
```

---

### 9.1 安全上传场景

条件：

```text
local.version > remote.version
local.lastSyncVersion == remote.version
vaultId 相同
密钥可以解密本地和云端
没有大量删除风险
```

说明本地是在当前云端版本基础上修改出来的。

处理：

```text
自动上传本地文件覆盖云端。
上传成功后，设置 local.lastSyncVersion = local.version。
```

示例：

```text
云端版本：9
本地上次同步版本：9
本地当前版本：10
```

说明：

```text
本地从云端 9 修改到 10，可以安全上传。
```

---

### 9.2 冲突场景

条件：

```text
local.version > remote.version
local.lastSyncVersion != remote.version
```

示例：

```text
云端版本：9
本地上次同步版本：7
本地当前版本：10
```

说明：

```text
本地从版本 7 修改到了 10。
但云端也从版本 7 变成了 9。
这说明本地和云端都发生过修改。
```

处理：

```text
不能自动上传。
进入冲突处理。
```

提示用户：

```text
本地和云端都发生过修改，无法自动判断应该保留哪一份。

请选择：
- 使用本地覆盖云端
- 备份本地后使用云端覆盖本地
- 取消同步
```

如果以后支持条目级合并，可以增加：

```text
- 尝试合并本地和云端数据
```

---

### 9.3 本地版本更新，但存在大量删除

条件：

```text
local.version > remote.version
但是本地条目数量明显少于云端
```

例如：

```text
云端有 100 条
本地只有 20 条
```

说明可能是用户误删、本地文件损坏、程序异常或同步异常。

处理：

```text
不能自动上传。
必须提示用户。
```

提示文案：

```text
检测到本地数据相比云端减少较多。

云端条目数量：100
本地条目数量：20

这可能是你主动删除了数据，也可能是误删或本地文件异常。
为了避免误删同步到云端，本次不会自动上传。

请选择：
- 确认使用本地数据覆盖云端
- 从云端恢复数据
- 备份本地后从云端恢复
- 取消同步
```

默认选项：

```text
取消同步
```

或者：

```text
从云端恢复
```

---

## 10. 云端版本大于本地版本

例如：

```text
local.version = 9
remote.version = 10
```

表示云端看起来比本地新。

同样不能直接下载，需要判断本地是否有未同步修改。

---

### 10.1 安全下载场景

条件：

```text
remote.version > local.version
local.version == local.lastSyncVersion
vaultId 相同
密钥可以解密云端
```

说明本地没有未同步修改，只是云端更新了。

处理：

```text
自动下载云端文件覆盖本地。
覆盖前可以备份本地文件。
下载成功后，设置 local.lastSyncVersion = remote.version。
```

示例：

```text
本地版本：9
本地上次同步版本：9
云端版本：10
```

说明：

```text
本地没有改动，云端更新了，可以安全下载。
```

---

### 10.2 本地也有修改

条件：

```text
remote.version > local.version
local.version != local.lastSyncVersion
```

说明本地也有未同步修改。

处理：

```text
不能自动下载。
进入冲突处理。
```

提示用户：

```text
云端版本比本地新，但本地也存在未同步修改。
如果直接下载云端文件，可能会覆盖本地修改。

请选择：
- 使用云端覆盖本地
- 使用本地覆盖云端
- 备份本地后使用云端覆盖本地
- 取消同步
```

默认选项：

```text
取消同步
```

---

## 11. 本地版本等于云端版本

例如：

```text
local.version = 10
remote.version = 10
```

这不一定代表内容一致。

需要比较：

```text
local.contentHash
remote.contentHash
```

---

### 11.1 hash 相同

处理：

```text
无需同步。
```

---

### 11.2 hash 不同

说明版本号一样，但内容不一样。

这可能是：

1. 多设备同时修改，版本号冲突
2. 某一端版本号没有正确递增
3. 文件被手动替换
4. 程序异常

处理：

```text
不能自动覆盖。
进入冲突处理。
```

提示用户：

```text
本地和云端版本号相同，但内容不同。
这可能是多设备修改或同步异常导致的冲突。

请选择：
- 使用本地覆盖云端
- 使用云端覆盖本地
- 备份本地后使用云端覆盖本地
- 取消同步
```

---

## 12. 删除检测规则

密码管理箱必须防止误删被同步到云端。

每次上传前，需要判断本地数据是否相比云端明显减少。

建议规则：

```text
删除数量 >= 5
或者
删除比例 >= 30%
```

满足任一条件就认为是高风险删除。

示例：

```text
云端有 50 条
本地只有 30 条
减少 20 条
删除比例 40%
```

处理：

```text
不自动上传。
提示用户确认。
```

提示文案：

```text
检测到本地数据相比云端减少较多。

云端条目数量：50
本地条目数量：30
减少数量：20

为了避免误删同步到云端，本次不会自动上传。

请选择：
- 确认这是我主动删除的，使用本地覆盖云端
- 从云端恢复数据
- 备份本地后从云端恢复
- 取消同步
```

---

## 13. 覆盖前备份规则

只要涉及覆盖，就应该尽量创建备份。

### 13.1 本地覆盖云端前

建议：

```text
先下载当前云端文件作为备份。
备份文件名包含时间戳。
再上传本地文件覆盖云端。
```

示例备份名：

```text
passvault_remote_backup_2026-06-06_153000.pvbak
```

---

### 13.2 云端覆盖本地前

建议：

```text
先备份当前本地文件。
再用云端文件覆盖本地。
```

示例备份名：

```text
passvault_local_backup_2026-06-06_153000.pvbak
```

---

## 14. 用户操作选项设计

遇到冲突、高风险删除、vaultId 不一致、密钥不一致等场景时，不要只给"确定 / 取消"。

建议提供以下操作：

```text
1. 使用本地覆盖云端
2. 使用云端覆盖本地
3. 备份本地后使用云端覆盖本地
4. 备份云端后使用本地覆盖云端
5. 重新输入密钥
6. 取消同步
```

默认选项应该是：

```text
取消同步
```

对于危险操作，需要二次确认。

例如：

```text
你正在使用本地文件覆盖云端文件。
该操作可能导致云端数据丢失。

建议先备份云端文件。
是否继续？
```

---

## 15. 自动上传条件

只有同时满足以下条件，才允许自动上传本地文件覆盖云端：

```text
1. 本地文件存在
2. 云端文件存在
3. 本地和云端 vaultId 相同
4. 当前密钥可以解密本地和云端
5. local.version > remote.version
6. local.lastSyncVersion == remote.version
7. local.contentHash != remote.contentHash
8. 未检测到大量删除风险
```

伪代码：

```text
if local.exists
and remote.exists
and local.vaultId == remote.vaultId
and canDecryptLocal
and canDecryptRemote
and local.version > remote.version
and local.lastSyncVersion == remote.version
and local.contentHash != remote.contentHash
and not dangerousDelete:
    uploadLocalToRemote()
    local.lastSyncVersion = local.version
else:
    requireUserDecision()
```

---

## 16. 自动下载条件

只有同时满足以下条件，才允许自动下载云端文件覆盖本地：

```text
1. 本地文件存在
2. 云端文件存在
3. 本地和云端 vaultId 相同
4. 当前密钥可以解密云端
5. remote.version > local.version
6. local.version == local.lastSyncVersion
7. local.contentHash != remote.contentHash
```

伪代码：

```text
if local.exists
and remote.exists
and local.vaultId == remote.vaultId
and canDecryptRemote
and remote.version > local.version
and local.version == local.lastSyncVersion
and local.contentHash != remote.contentHash:
    backupLocal()
    downloadRemoteToLocal()
    local.lastSyncVersion = remote.version
else:
    requireUserDecision()
```

---

## 17. 禁止自动处理的场景

以下场景必须禁止自动同步：

```text
1. 当前密钥无法解密云端文件
2. 当前密钥无法解密本地文件
3. 本地和云端 vaultId 不一致
4. 本地和云端版本相同但 hash 不同
5. 本地和云端都发生过修改
6. 本地存在大量删除风险
7. 云端文件疑似损坏
8. 本地文件疑似损坏
9. 用户第一次启用云同步且本地、云端都有数据
```

这些场景都需要用户手动选择。

---

## 18. 首次启用云同步规则

如果用户之前一直使用本地文件，后来才开启坚果云同步，需要特殊处理。

### 18.1 本地有文件，云端无文件

提示：

```text
云端没有发现同步文件。
是否将当前本地密码库上传到云端？
```

用户确认后上传。

---

### 18.2 本地无文件，云端有文件

提示：

```text
检测到云端已有密码库。
请输入云端密码库的密钥以恢复数据。
```

密钥正确后下载。

---

### 18.3 本地有文件，云端也有文件

不能直接比较版本号。

先判断：

```text
1. 密钥是否能解密本地和云端
2. vaultId 是否一致
3. 内容 hash 是否一致
4. 版本号和 lastSyncVersion 是否能证明同步关系
```

如果 vaultId 不一致，提示：

```text
检测到本地密码库和云端密码库不是同一个库。
请确认你要使用哪一份数据。

请选择：
- 使用本地覆盖云端
- 备份本地后使用云端覆盖本地
- 取消同步
```

默认取消。

---

## 19. 密钥迁移规则

如果用户输入旧云端密钥后成功解密云端文件，可以提供密钥迁移功能。

流程：

```text
1. 用户输入旧云端密钥
2. 程序成功解密云端文件
3. 用户输入新的本地密钥
4. 使用新密钥重新加密数据
5. 上传新加密后的文件到云端
6. 本地也切换到新密钥
```

如果第一版不想做复杂，可以先不支持密钥迁移，只提示：

```text
请使用云端密码库对应的旧密钥登录。
```

---

## 20. 推荐同步流程

完整流程如下：

```text
开始同步
  ↓
检查本地文件是否存在
检查云端文件是否存在
  ↓
本地无 + 云端无
  → 创建新密码库
  → 如果开启云同步，则上传云端

本地有 + 云端无
  → 提示是否上传本地到云端
  → 用户确认后上传

本地无 + 云端有
  → 要求输入云端密钥
  → 解密成功后下载云端

本地有 + 云端有
  ↓
  验证密钥
  ↓
  判断 vaultId
  ↓
  判断 contentHash
  ↓
  判断 version
  ↓
  判断 lastSyncVersion
  ↓
  判断大量删除风险
  ↓
  能安全同步 → 自动上传或下载
  不能安全同步 → 提示用户选择
```

---

## 21. 最终结论

PassVault 的坚果云同步不能只依赖版本号。

正确的同步判断应该由以下信息共同决定：

```text
vaultId：判断是不是同一个密码库
密钥验证：判断能不能解密
version：判断数据新旧
lastSyncVersion：判断是否存在双端修改
contentHash：判断内容是否真的一致
itemCount / deletedItems：判断是否存在误删风险
备份机制：防止覆盖后无法恢复
```

最终规则：

```text
本地版本高，不一定直接上传。
云端版本高，不一定直接下载。
密钥不一致，禁止自动解析和覆盖。
vaultId 不一致，说明不是同一个库，必须让用户选择。
大量删除时，不自动上传，必须提示用户确认。
任何覆盖操作前，优先备份。
```

最安全的策略是：

```text
安全场景自动同步；
危险场景提示用户；
密钥不一致停止同步；
覆盖前先备份。
```
